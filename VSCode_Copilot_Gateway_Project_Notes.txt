During Development ====================================================

GitHub Copilot

Built-in GitHub Copilot Chat with support for Custom Endpoints.

Ollama VS Code Extension

Not used.

Architecture:

VS Code | Django Gateway | OpenRouter (Current) | Future -> Ollama

==================================================== Objective
====================================================

Build a Django REST Framework gateway that allows VS Code GitHub Copilot
Chat (Custom Endpoint) to communicate with any LLM provider through a
single OpenAI-compatible endpoint.

Current Architecture:

VS Code Copilot Chat | v Custom Endpoint | v Django REST API Gateway
(/v1/chat/completions) | v OpenRouter (Current) | v LLM (NVIDIA Nemotron
/ Cohere during testing depending on model selected)

Future Architecture:

VS Code Copilot Chat | v Django REST API Gateway | +–> OpenRouter +–>
Ollama (Gemma 4) +–> Azure OpenAI +–> Gemini +–> Any OpenAI Compatible
Provider

==================================================== Project Goal
====================================================

The goal is to make VS Code communicate ONLY with Django.

Django becomes the single gateway responsible for:

-   Authentication
-   Authorization
-   Logging
-   Prompt Processing
-   Provider Selection
-   RAG (future)
-   Conversation Storage
-   Analytics
-   Rate Limiting
-   Cost Tracking
-   Model Routing

VS Code should never directly communicate with OpenRouter or Ollama.

==================================================== VS Code Custom
Endpoint Configuration
====================================================

The following file is the MOST IMPORTANT configuration inside VS Code.

Filename:

chatLanguageModels.json

Current configuration used during development:

[ { “name”: “Custom DRF Gateway v2”, “vendor”: “customendpoint”,
“apiKey”: “dummy”, “apiType”: “chat-completions”, “models”: [ { “id”:
“cohere/north-mini-code:free”, “name”: “cohere Free”, “url”:
“http://127.0.0.1:8000/v1/chat/completions/”, “toolCalling”: true,
“vision”: true, “maxInputTokens”: 128000, “maxOutputTokens”: 16000 } ]
}]

Meaning of every field

name Friendly provider name shown inside VS Code.

vendor Must always be “customendpoint”.

apiKey VS Code automatically sends this value as

    Authorization: Bearer <apiKey>

    During testing we used:
        dummy

    Future:
        JWT Access Token from Django Login.

apiType We selected: chat-completions

    This means VS Code expects an OpenAI Chat Completions compatible API.

models List of models exposed by the gateway.

id Value sent inside the request payload as:

        "model": "<id>"

name Friendly model name shown inside VS Code.

url Django REST endpoint.

toolCalling Enables OpenAI Tool Calling support.

vision Advertises vision capability.

maxInputTokens Maximum supported input context.

maxOutputTokens Maximum supported output tokens.

==================================================== Implementation
Completed ====================================================

1.  Created Django Project.

2.  Installed:

-   Django
-   Django REST Framework
-   httpx
-   python-dotenv

3.  Created endpoint:

POST /v1/chat/completions/

4.  Stored OpenRouter API key inside .env

5.  Implemented OpenAI Chat Completions compatible proxy.

6.  Verified endpoint using Thunder Client.

7.  Configured VS Code Custom Endpoint.

8.  Observed that VS Code sends an OpenAI Chat Completions request.

Payload contains fields such as:

-   model
-   messages
-   tools
-   tool_choice
-   temperature
-   top_p
-   stream

9.  Initial implementation returned JSON using DRF Response().

Thunder Client worked.

VS Code failed because it expects streaming.

10. Implemented Streaming Proxy using:

-   httpx
-   StreamingHttpResponse

The gateway now forwards streaming bytes from OpenRouter without
modifying them.

11. Result:

VS Code -> Django -> OpenRouter works successfully.

==================================================== Important Lessons
Learned ====================================================

1.  Never change request payload unless required.

Forward everything exactly as received.

2.  Never force:

payload[“stream”] = False

VS Code sends stream=True.

3.  Use streaming mode with OpenRouter.

4.  Forward stream using Django StreamingHttpResponse.

5.  Keep upstream connection alive until streaming finishes.

6.  Do not replace StreamingHttpResponse with DRF Response().

7.  The gateway should behave like a transparent OpenAI-compatible
    proxy.

==================================================== Current Request
Flow ====================================================

VS Code | POST /v1/chat/completions | Django Gateway

-   Receive request

-   Read Authorization header
-   Read JSON payload
-   Forward payload unchanged
-   Stream response back

OpenRouter

-   Generates streamed response

-   Returns stream unchanged using StreamingHttpResponse

VS Code

-   Displays streamed response.

==================================================== Important Notes for
Continuation ====================================================

The current views.py already implements the streaming proxy.

Do NOT replace it with a normal DRF Response().

The gateway must continue using StreamingHttpResponse for compatibility
with VS Code Custom Endpoint.

The project philosophy is:

VS Code -> Django Gateway -> Provider

instead of

VS Code -> Provider

The gateway should remain provider-agnostic so that only one provider
module changes when migrating from OpenRouter to Ollama.

IMPORTANT: Any future implementation (JWT, logging, RAG, provider
routing, analytics, cost tracking, etc.) should preserve the OpenAI Chat
Completions API contract so that no changes are required inside VS Code.
